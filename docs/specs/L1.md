# L1 - High-Level Requirements

The Upper Room is a mobile-first web application that enables City Leads of a faith-based technology organization to coordinate their local and global teams, manage partner organizations, host hackathons following the FaithTech 4 D's process, and track contacts and outreach activities. These high-level requirements define what the system must do across functional and cross-cutting concerns.

---

## L1-001: User Authentication
The system must authenticate users before granting access to any application functionality, supporting account creation, sign-in, sign-out, password recovery, and session lifecycle management.

## L1-002: Role-Based Authorization
The system must enforce role-based access control (RBAC) so that users can only perform actions permitted by their assigned role(s), including the four organizational roles (Prayer Lead, City Lead, Event Lead, Communication Lead) and a system Administrator role.

## L1-003: Contact Management
The system must allow authorized users to create, view, update, delete, search, and annotate contacts with free-form notes, where each contact represents an individual person tracked by the organization.

## L1-004: Partner Management
The system must allow authorized users to manage partner organizations through a sales-style funnel (Lead, In Funnel, Confirmed Partner), associate one or more contacts with each partner, and attach free-form notes to each partner record.

## L1-005: Hackathon Management
The system must allow authorized users to plan, run, and document hackathons that are co-hosted with partner organizations, with each hackathon producing one or more software products by following the FaithTech 4 D's process (Discover, Design, Develop, Deploy).

## L1-006: Team Management
The system must allow authorized users to manage their local team's membership and role assignments (Prayer Lead, City Lead, Event Lead, Communication Lead) and to view and discover other teams across all cities globally.

## L1-007: Configurable Dashboard
The system must provide each user with a personalized dashboard composed of resizable, draggable widgets that display real-time metrics, charts, and summaries relevant to that user's role and team.

## L1-008: Real-Time Updates and Notifications
The system must push live updates, notifications, and metric refreshes to connected clients without requiring a page reload, so that users see changes (new contacts, partner stage changes, team activity, hackathon updates) as they happen.

## L1-009: Responsive Mobile-First Design
The system must render correctly and remain fully usable across all viewport sizes, from extra-small mobile screens (<576px) up to extra-large desktop displays (≥1200px), with mobile as the primary design target.

## L1-010: Dark Monochromatic Theming
The system must present a dark, mostly monochromatic visual theme as its default and only theme, applied consistently across every screen, component, and chart.

## L1-011: Cross-Entity Search
The system must allow users to search across contacts, partners, hackathons, and team members from a single search affordance, returning relevant results categorized by entity type.

## L1-012: Logging and Observability
The system must produce structured logs on both the frontend and the backend that capture user actions, system events, errors, and performance metrics, sufficient for debugging, auditing, and operational monitoring.

## L1-013: Performance and Scalability
The system must meet defined performance targets for page load, API response times, and real-time message delivery, and must scale to support the expected volume of users, contacts, partners, and concurrent connections.

## L1-014: Data Protection and Application Security
The system must protect user data and application integrity by securing data in transit and at rest, validating all inputs, defending against the OWASP Top 10 vulnerabilities, and enforcing rate limits on sensitive endpoints.

## L1-015: Accessibility
The system must meet WCAG 2.1 Level AA accessibility standards so that users relying on assistive technologies, keyboard navigation, or high-contrast displays can use every feature.

## L1-016: Reusable Component Library
The system must provide a frontend component library that wraps Angular Material primitives into the application's branded, dark, monochromatic design language, so that every feature reuses the same buttons, dialogs, error messages, notifications, form fields, and other presentation components.

## L1-017: API Contract and Decoupled Integration
The system must expose a stable backend API and provide a frontend API library that defines models and service contracts so feature libraries depend only on injected service tokens and never on concrete service implementations.

## L1-018: End-to-End Test Coverage
The system must include automated end-to-end tests that exercise every major user flow through the running frontend and backend, written using the Page Object Model pattern.
