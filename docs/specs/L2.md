# L2 - Detailed Requirements

Each L2 requirement traces to exactly one L1 requirement. Acceptance criteria are written in Given/When/Then form so that they translate directly into executable Playwright end-to-end tests, backend integration tests, or Angular component tests.

When acceptance tests are written, every test file must include a header comment identifying the L2 requirement(s) it covers, for example:

```typescript
// Acceptance Test
// Traces to: L2-009, L2-010
// Description: Verify a City Lead can create and view a contact
```

---

## Authentication

### L2-001: User Registration
**Traces to:** L1-001

The system must allow a new user to register an account by submitting an email address, a password, a display name, and the city the user belongs to. Registration must require email verification before the account can be used.

**Acceptance Criteria:**
1. Given an unauthenticated visitor on the registration screen, when they submit a valid email, password (≥12 characters with mixed case, digit, and symbol), display name, and city, then the system must create a pending account and send a verification email within 30 seconds.
2. Given a pending account, when the user clicks the verification link within 24 hours of registration, then the account must become active and the user must be redirected to the sign-in screen.
3. Given an email address that is already registered, when a visitor submits the registration form with that email, then the system must respond with a generic confirmation message (so the existence of the account is not disclosed) and must not create a duplicate account.
4. Given any registration submission, when the password fails the strength rules, then the system must reject the submission with a field-level validation message and must not create a pending account.

---

### L2-002: User Sign-In
**Traces to:** L1-001

The system must authenticate a user by verified email and password, issue a session, and redirect the user to their dashboard.

**Acceptance Criteria:**
1. Given an active account, when the user submits the correct email and password, then the system must issue an authenticated session and redirect the user to the dashboard within 2 seconds.
2. Given an unverified account, when the user submits correct credentials, then the system must reject the sign-in with a message indicating verification is required.
3. Given any sign-in form submission, when the credentials are invalid, then the system must respond with a generic "invalid email or password" message and must not disclose which field failed.
4. Given five consecutive failed sign-in attempts for the same email within 15 minutes, when a sixth attempt is made, then the system must lock further attempts for that email for 15 minutes and log the lockout event.

---

### L2-003: Password Recovery
**Traces to:** L1-001

The system must allow a user to reset their password through an email-based recovery flow.

**Acceptance Criteria:**
1. Given an unauthenticated visitor on the password recovery screen, when they submit a registered email, then the system must send a single-use recovery link to that email within 30 seconds and must always display the same generic "if the email exists, a link has been sent" message.
2. Given a recovery link issued less than 60 minutes ago, when the user opens it and submits a new password meeting strength rules, then the system must update the password, invalidate all existing sessions for that account, and require sign-in.
3. Given a recovery link that is older than 60 minutes or has already been used, when the user opens it, then the system must reject the link with an "expired or already used" message.

---

### L2-004: Session Management
**Traces to:** L1-001

The system must manage authenticated sessions using HTTP-only, Secure, SameSite cookies and must support sliding expiration with an absolute maximum lifetime.

**Acceptance Criteria:**
1. Given a successful sign-in, when the system issues the session cookie, then the cookie must have HttpOnly, Secure, and SameSite=Lax (or stricter) attributes set.
2. Given an active session, when the user makes any authenticated request, then the session expiration must be extended by 30 minutes (sliding) up to an absolute maximum of 12 hours from sign-in.
3. Given a session that has reached its absolute maximum lifetime, when the user makes any authenticated request, then the system must reject the request with 401 Unauthorized and require a fresh sign-in.

---

### L2-005: User Sign-Out
**Traces to:** L1-001

The system must allow a user to sign out from any screen, ending the session on the server and clearing the client-side session.

**Acceptance Criteria:**
1. Given an authenticated user, when the user activates the sign-out control, then the system must invalidate the session on the backend, clear the session cookie, close any open SignalR connection, and redirect to the sign-in screen within 1 second.
2. Given a signed-out session, when an attacker replays the previous session cookie, then the backend must reject the request with 401 Unauthorized.

## Authorization

### L2-006: Role Definitions and Assignment
**Traces to:** L1-002

The system must define five roles — Administrator, City Lead, Prayer Lead, Event Lead, Communication Lead — and must allow Administrators and City Leads to assign roles to users on their team.

**Acceptance Criteria:**
1. Given an Administrator, when they assign any of the five roles to any user, then the assignment must persist and take effect on the user's next request without requiring sign-out.
2. Given a City Lead, when they assign Prayer Lead, Event Lead, or Communication Lead roles to a member of their local team, then the assignment must succeed; when they attempt to assign Administrator or City Lead, then the system must reject the request with 403 Forbidden.
3. Given any other role, when the user attempts to assign any role, then the system must reject the request with 403 Forbidden.

---

### L2-007: Endpoint Authorization Enforcement
**Traces to:** L1-002

Every backend endpoint must enforce a documented authorization policy, and every Angular route must enforce the same policy via guards.

**Acceptance Criteria:**
1. Given an unauthenticated request to any endpoint other than the public authentication endpoints, when the request is received, then the backend must respond with 401 Unauthorized.
2. Given an authenticated user without the required role, when the user calls a protected endpoint, then the backend must respond with 403 Forbidden and must log the denial with user ID, endpoint, and timestamp.
3. Given an authenticated user without the required role, when the user navigates to a protected Angular route, then the route guard must redirect to a "no access" screen and must not load the underlying feature module.

---

### L2-008: Cross-Team Data Isolation
**Traces to:** L1-002

A user's read and write access to team-scoped data (contacts, partners, hackathons, team members, notes) must be limited to their own local team unless they are an Administrator.

**Acceptance Criteria:**
1. Given a non-Administrator user, when they query, create, update, or delete any contact, partner, hackathon, or note, then the backend must reject any operation that targets a record outside the user's local team with 403 Forbidden or 404 Not Found.
2. Given an Administrator, when they perform the same operations, then the backend must allow the operation regardless of team boundary.
3. Given a global team listing request from a non-Administrator (per L2-033), then the backend must return only public team summary fields (city, role counts, active hackathon count) and must not return contacts, partner notes, or other private records.

## Contact Management

### L2-009: Create Contact
**Traces to:** L1-003

The system must allow City Leads, Prayer Leads, Event Leads, and Communication Leads to create a contact record with first name, last name, email (optional), phone (optional), city (optional), and free-form notes (optional).

**Acceptance Criteria:**
1. Given an authorized user on the new-contact form, when they submit a record with at least first name and last name, then the system must persist the contact, attribute it to the user's local team, and return the new contact's ID within 1 second.
2. Given a submitted email address, when the email is malformed, then the system must reject the submission with a field-level validation message.
3. Given a submitted phone number, when the phone format does not match a valid international E.164 or local format, then the system must reject the submission with a field-level validation message.
4. Given any required field is missing, when the user submits the form, then the submission must be blocked client-side and rejected server-side with a 400 Bad Request response.

---

### L2-010: View Contact
**Traces to:** L1-003

The system must display a contact's full details, including all notes ordered most recent first.

**Acceptance Criteria:**
1. Given an authorized user, when they open a contact assigned to their team, then the system must display first name, last name, email, phone, city, creation date, last-updated date, the user who created the contact, and all notes ordered with the most recent first.
2. Given a non-Administrator user, when they request a contact belonging to a different team, then the system must return 404 Not Found (per L2-008).
3. Given a viewport width <576px, when the user views the contact detail screen, then the layout must use a single-column stack with notes appearing below the contact summary.

---

### L2-011: Update Contact
**Traces to:** L1-003

The system must allow authorized users to update any field of a contact in their team.

**Acceptance Criteria:**
1. Given an authorized user editing a contact, when they save changes that pass validation, then the system must persist the changes, update the last-updated timestamp, record the modifying user, and reflect the new values within 1 second.
2. Given two users editing the same contact concurrently, when the second save is submitted with stale data, then the system must detect the conflict (via ETag or version field) and reject the second save with a 409 Conflict response and a clear error.

---

### L2-012: Delete Contact
**Traces to:** L1-003

The system must allow City Leads and Administrators to delete a contact from their team. Other roles must not delete contacts.

**Acceptance Criteria:**
1. Given a City Lead or Administrator, when they delete a contact, then the system must require an explicit confirmation dialog, perform a soft delete (set deleted-at timestamp), and remove the contact from default lists and search results.
2. Given a Prayer Lead, Event Lead, or Communication Lead, when they attempt to delete a contact, then the system must reject the request with 403 Forbidden.
3. Given a soft-deleted contact, when an Administrator queries the audit/restore view, then the contact must be visible there and must be restorable.

---

### L2-013: Add and Edit Notes on Contact
**Traces to:** L1-003

The system must allow authorized users to add free-form text notes (up to 4000 characters per note) to any contact in their team and to edit or delete notes they themselves authored.

**Acceptance Criteria:**
1. Given an authorized user on a contact detail screen, when they submit a note of 1–4000 characters, then the system must persist the note with author, timestamp, and association to the contact, and must display it at the top of the notes list immediately.
2. Given a user, when they edit or delete a note they authored, then the system must allow the operation; when they attempt to edit or delete a note authored by another user, then the system must reject the operation with 403 Forbidden, except for Administrators and City Leads who may edit or delete any note in their team.
3. Given a note that exceeds 4000 characters, when the user submits it, then the system must reject the submission with a validation message.

---

### L2-014: Search Contacts
**Traces to:** L1-003, L1-011

The system must provide search across contacts by first name, last name, email, phone, and note content within the user's team scope.

**Acceptance Criteria:**
1. Given an authorized user, when they enter a search term of 2 or more characters in the contact search field, then the system must return contacts whose first name, last name, email, phone, or any note contains the term (case-insensitive) within 500 ms for a dataset of up to 10,000 contacts.
2. Given a search term of fewer than 2 characters, when submitted, then the system must not execute the search.
3. Given search results, when displayed, then each result must show first name, last name, city, and a snippet of the matched field with the term highlighted.

---

### L2-015: List and Paginate Contacts
**Traces to:** L1-003

The system must list a team's contacts with pagination, sorting, and basic filtering.

**Acceptance Criteria:**
1. Given an authorized user, when they open the contacts list, then the system must display contacts in pages of 25, sorted by last name ascending by default.
2. Given the contacts list, when the user clicks a sortable column header, then the list must re-sort by that column and toggle ascending/descending on subsequent clicks.
3. Given a viewport width <576px, when the contacts list is rendered, then it must display as a vertical card list rather than a table, and the sort and filter controls must collapse into a menu.

## Partner Management

### L2-016: Create Partner Organization
**Traces to:** L1-004

The system must allow authorized users to create a partner organization with name, website (optional), city, current funnel stage (Lead, In Funnel, Confirmed), and an optional description.

**Acceptance Criteria:**
1. Given an authorized user, when they submit a new partner with at least name and stage, then the system must persist the partner, default the stage to "Lead" when not provided, attribute it to the user's team, and return the partner's ID within 1 second.
2. Given a submitted website URL, when the URL is malformed, then the system must reject the submission with a field-level validation message.

---

### L2-017: Move Partner Through Funnel Stages
**Traces to:** L1-004

The system must allow authorized users to change a partner's funnel stage among Lead, In Funnel, and Confirmed Partner, and must record every stage change with timestamp and acting user.

**Acceptance Criteria:**
1. Given an authorized user, when they change a partner's stage, then the system must persist the new stage, append a stage-change history entry (from-stage, to-stage, user, timestamp), and broadcast the change via SignalR to other connected team members within 2 seconds.
2. Given the partner detail screen, when displayed, then the partner's full stage history must be viewable in chronological order.

---

### L2-018: Associate Contacts with Partner
**Traces to:** L1-004

Each partner must have zero or more contacts associated with it, where each contact represents a person at the partner organization.

**Acceptance Criteria:**
1. Given a partner detail screen, when an authorized user adds an existing contact (per L2-009) as a contact of the partner, then the system must persist the association and display the contact in the partner's contact list within 1 second.
2. Given a partner detail screen, when the user creates a new contact directly from the partner screen, then the system must create the contact (per L2-009) and associate it with the partner in a single submission.
3. Given an associated contact, when the user removes the association from the partner, then the contact record must remain in the team's contact list but must no longer appear under that partner.

---

### L2-019: Add Notes to Partner
**Traces to:** L1-004

The system must allow authorized users to add free-form notes (up to 4000 characters per note) to any partner in their team. Authoring, edit, and delete rules match L2-013.

**Acceptance Criteria:**
1. Given an authorized user on a partner detail screen, when they submit a note of 1–4000 characters, then the system must persist the note with author, timestamp, and association to the partner, and display it at the top of the partner's notes list immediately.
2. Given a user, when they edit or delete a note they authored on a partner, then the system must allow the operation; when they attempt to edit or delete another user's note, then the system must reject with 403 Forbidden, except for Administrators and City Leads.

---

### L2-020: View, Update, and Delete Partner
**Traces to:** L1-004

The system must allow authorized users to view, update, and delete partner records under the same authorization model as contacts (delete restricted to City Lead and Administrator).

**Acceptance Criteria:**
1. Given an authorized user, when they open a partner in their team, then the system must display name, website, city, current stage, stage history, all associated contacts, and all notes most-recent-first.
2. Given a City Lead or Administrator, when they delete a partner, then the system must require explicit confirmation, soft-delete the partner, and detach (but not delete) the associated contacts.
3. Given a Prayer Lead, Event Lead, or Communication Lead, when they attempt to delete a partner, then the system must reject the request with 403 Forbidden.

---

### L2-021: Filter Partners by Funnel Stage
**Traces to:** L1-004

The system must allow users to view the partner list filtered by funnel stage and to view a Kanban-style board grouped by stage.

**Acceptance Criteria:**
1. Given the partner list screen, when the user selects one or more stage filters (Lead, In Funnel, Confirmed), then only partners in the selected stages must be displayed.
2. Given the partner board screen on a viewport ≥768px, when the user views it, then partners must be displayed as cards in three columns (Lead, In Funnel, Confirmed) with the card count visible per column.
3. Given a viewport <768px, when the user views the partner board, then the board must collapse to a single-column stack with stage section headers, preserving all card content.

## Hackathon Management

### L2-022: Create Hackathon
**Traces to:** L1-005

The system must allow authorized users to create a hackathon with title, start date, end date, host city, one or more associated partners, and an initial 4 D's stage of "Discover".

**Acceptance Criteria:**
1. Given an authorized user, when they submit a new hackathon with title, start date, and end date (start ≤ end), then the system must persist the hackathon, default the 4 D's stage to "Discover", and return the hackathon's ID within 1 second.
2. Given an end date that precedes the start date, when the user submits the form, then the system must reject the submission with a field-level validation message.
3. Given an authorized user, when they associate one or more existing partners with the hackathon at creation time or afterward, then the system must persist each association.

---

### L2-023: Track 4 D's Process Stage
**Traces to:** L1-005

The system must allow authorized users to advance a hackathon through the FaithTech 4 D's stages (Discover → Design → Develop → Deploy) and must record every stage change.

**Acceptance Criteria:**
1. Given a hackathon, when an authorized user advances or retreats the stage among the four D's, then the system must persist the new stage, append a stage-history entry (from-stage, to-stage, user, timestamp), and broadcast the change via SignalR to other connected team members.
2. Given a hackathon detail screen, when displayed, then the four D's must be shown as a visual progress indicator with the current stage highlighted and the full stage history available.

---

### L2-024: Document Hackathon Products
**Traces to:** L1-005

Each hackathon must support zero or more product records, where each product represents a software output of the hackathon, with name, description, repository URL (optional), demo URL (optional), and the team members who built it.

**Acceptance Criteria:**
1. Given a hackathon detail screen, when an authorized user adds a product with at least a name, then the system must persist the product associated with the hackathon and display it within 1 second.
2. Given a product with a repository or demo URL, when the URL is malformed, then the system must reject the submission with a field-level validation message.
3. Given an authorized user, when they assign team members (existing users or contacts) to a product, then the system must persist the assignments and display them on the product card.

---

### L2-025: View Hackathon Details
**Traces to:** L1-005

The system must display a hackathon's full details, including dates, host city, associated partners, current 4 D's stage with history, and all product records.

**Acceptance Criteria:**
1. Given an authorized user, when they open a hackathon in their team, then the system must display all details listed above, ordered with products first, then partners, then stage history.
2. Given a viewport <576px, when the user views the hackathon detail screen, then the layout must use a single-column stack with collapsible sections for products, partners, and stage history.

---

## Team Management

### L2-026: View Local Team
**Traces to:** L1-006

The system must display the user's local team's members, each with display name, role(s), email, and active status.

**Acceptance Criteria:**
1. Given an authenticated user, when they open the local team screen, then the system must display all members of their local team with name, role(s), email, and active status, sorted by role then last name.
2. Given a viewport <576px, when the team screen is rendered, then members must be displayed as a vertical card list rather than a table.

---

### L2-027: Add Team Member
**Traces to:** L1-006

The system must allow City Leads and Administrators to add a user to the local team by email invitation, optionally with one or more roles assigned at invitation time.

**Acceptance Criteria:**
1. Given a City Lead or Administrator, when they submit an invitation with an email address and zero or more roles, then the system must send an invitation email containing a single-use link valid for 7 days.
2. Given the invitation link, when the recipient completes registration (per L2-001), then the system must add the new user to the inviter's local team and assign the roles specified in the invitation.
3. Given a Prayer Lead, Event Lead, or Communication Lead, when they attempt to invite a team member, then the system must reject the request with 403 Forbidden.

---

### L2-028: Remove Team Member
**Traces to:** L1-006

The system must allow City Leads and Administrators to remove a user from the local team.

**Acceptance Criteria:**
1. Given a City Lead or Administrator, when they remove a member from their local team, then the system must require explicit confirmation, revoke the user's roles on that team, end any active session that user has, and broadcast the change via SignalR.
2. Given a City Lead, when they attempt to remove another City Lead or an Administrator, then the system must reject the request with 403 Forbidden; only Administrators may remove these higher-privilege roles.

---

### L2-029: Assign Team Role
**Traces to:** L1-006

The system must allow City Leads and Administrators to assign or revoke the four organizational roles (Prayer Lead, City Lead, Event Lead, Communication Lead) on members of the local team, subject to the constraints in L2-006.

**Acceptance Criteria:**
1. Given a City Lead, when they assign or revoke Prayer Lead, Event Lead, or Communication Lead on a local team member, then the system must persist the change and broadcast it via SignalR.
2. Given a City Lead, when they attempt to assign or revoke City Lead or Administrator roles, then the system must reject the request with 403 Forbidden.
3. Given any user, when their role assignments change, then the change must take effect on their next request without requiring sign-out.

---

### L2-030: View Global Teams
**Traces to:** L1-006

The system must allow any authenticated user to view a list of all teams across all cities and to open a public summary view of any team.

**Acceptance Criteria:**
1. Given an authenticated user, when they open the global teams screen, then the system must display every team with city, member count by role, active hackathon count, and partner count, paginated 25 per page.
2. Given a non-Administrator user, when they open another team's summary, then the system must display only public summary fields (city, role counts, active hackathon count, partner count) and must not expose any contact data, partner notes, or contact notes.
3. Given an Administrator, when they open any team, then the system must display the full team detail including contacts, partners, and notes.

---

### L2-031: Search Global Teams
**Traces to:** L1-006, L1-011

The system must allow users to search global teams by city or team member display name.

**Acceptance Criteria:**
1. Given an authenticated user, when they enter a search term of 2 or more characters on the global teams screen, then the system must return teams whose city or member display name matches (case-insensitive) within 500 ms.
2. Given a search term of fewer than 2 characters, when submitted, then the system must not execute the search.

## Dashboard

### L2-032: Default Dashboard Layout
**Traces to:** L1-007

The system must provide every newly registered user with a default dashboard composed of an empty placeholder area into which the user can drop widgets.

**Acceptance Criteria:**
1. Given a newly registered user signing in for the first time, when they reach the dashboard, then the system must display an empty dashboard area with a clear "Add widget" call to action.
2. Given a returning user, when they sign in, then the dashboard must restore the layout, position, and size of every widget exactly as last saved by that user.

---

### L2-033: Add, Resize, and Reposition Widgets
**Traces to:** L1-007

The system must allow each user to add widgets to the dashboard from a widget catalog, drag widgets to reposition them, and resize widgets within the dashboard grid.

**Acceptance Criteria:**
1. Given the dashboard, when the user opens the widget catalog and selects a widget, then the widget must be added to the next available grid cell and persisted within 1 second.
2. Given a widget on the dashboard, when the user drags it to a new position, then the new position must be persisted on drop and reflected on the dashboard within 500 ms.
3. Given a widget on the dashboard, when the user resizes it via the resize handle, then the new size must be persisted on resize-end and the widget content must reflow within the new dimensions.
4. Given a widget on the dashboard, when the user removes it, then the widget must disappear within 500 ms and the layout must persist.

---

### L2-034: Real-Time Metric Charts
**Traces to:** L1-007, L1-008

The dashboard widget catalog must include line-chart widgets that render team metrics (e.g., new contacts per day, partner stage transitions per week, hackathon stage progress) using Chart.js, refreshed in real time without page reload.

**Acceptance Criteria:**
1. Given a line-chart widget on the dashboard, when the user opens it, then the widget must render the chart with the most recent data within 1 second.
2. Given the user is viewing a line-chart widget, when underlying data changes (a new contact is created, a partner stage moves, etc.), then the chart must update within 2 seconds via SignalR push without a full page reload.
3. Given a viewport <576px, when a line-chart widget is rendered, then the chart must remain legible — axes labeled, legend wrapped, and touch interaction (tap to view a data point) supported.

---

### L2-035: Persisted Dashboard Layout per User
**Traces to:** L1-007

The system must persist each user's dashboard layout (widget set, position, size, configuration) per user.

**Acceptance Criteria:**
1. Given a user with a customized dashboard, when they sign out and sign back in (on the same or a different device), then the dashboard must restore exactly as left.
2. Given concurrent dashboard edits from the same user on two devices, when both submit layout changes, then the most-recent submission must win (last-write-wins) and the system must not error.

## Real-Time Updates and Notifications

### L2-036: Real-Time Connection Lifecycle
**Traces to:** L1-008

The system must establish a SignalR connection for every authenticated user upon application load and must reconnect automatically with exponential backoff if the connection drops.

**Acceptance Criteria:**
1. Given an authenticated user loading any application screen, when the application bootstraps, then the SignalR connection must be established within 2 seconds and the user must be subscribed to their team's channel.
2. Given an active SignalR connection, when the connection drops, then the client must attempt to reconnect with exponential backoff starting at 1 second, doubling up to a maximum of 60 seconds, indefinitely until success or sign-out.
3. Given the user signs out, when sign-out completes, then the SignalR connection must be closed and any reconnection attempts must stop.

---

### L2-037: Push Notifications for Team Events
**Traces to:** L1-008

The system must push real-time notifications to all connected members of a team when significant events occur on that team's records: contact created, partner stage changed, hackathon stage changed, new note added, team member added or removed.

**Acceptance Criteria:**
1. Given two team members, A and B, both signed in and connected, when A creates a contact, changes a partner stage, advances a hackathon stage, adds a note, or A is added/removed by an admin, then B's client must receive a SignalR message with event type, entity ID, actor ID, and timestamp within 2 seconds.
2. Given B's client receives a notification, when the event affects a screen B is currently viewing (e.g., the partner detail screen for the changed partner), then that screen must update its displayed data within 500 ms of receiving the notification, without a full page reload.

---

### L2-038: Notification Center
**Traces to:** L1-008

The system must persist notifications per user and must provide a notification center accessible from every screen, showing unread and read notifications.

**Acceptance Criteria:**
1. Given a notification pushed to a user, when the user is offline, then the notification must be persisted and delivered upon next sign-in via the notification center.
2. Given the notification center, when the user opens it, then unread notifications must appear at the top with a visible unread indicator; the unread count must be displayed on the notification center icon at all times.
3. Given a notification, when the user clicks it, then the notification must be marked as read, the unread count must decrement, and the user must navigate to the related entity screen.
4. Given the notification center, when the user activates "Mark all as read", then every unread notification must be marked as read within 500 ms.

## Responsive Design

### L2-039: Extra-Small Viewport Layout (<576px)
**Traces to:** L1-009

Every screen must render correctly and remain fully functional on viewports narrower than 576px, with single-column layouts, collapsible navigation, and touch-friendly hit targets.

**Acceptance Criteria:**
1. Given any application screen, when the viewport is <576px wide, then all primary content must use a single-column stack, the side navigation must collapse into a hamburger menu, and no horizontal scroll bar must appear except inside scrollable widgets like tables or charts that opt-in.
2. Given any interactive control (button, link, form input, list item), when rendered on <576px, then the control's hit target must be at least 44×44 CSS pixels.

---

### L2-040: Small to Medium Viewport Layout (≥576px and <992px)
**Traces to:** L1-009

Screens must scale gracefully across small (≥576px) and medium (≥768px) viewports, using two-column layouts where appropriate.

**Acceptance Criteria:**
1. Given any list screen (contacts, partners, hackathons, team), when the viewport is ≥576px and <768px, then the layout must remain a single-column card list but row density may increase.
2. Given any detail screen, when the viewport is ≥768px and <992px, then the layout may use a two-column split with primary content on the left and metadata/notes on the right.
3. Given the partner board screen (per L2-021), when the viewport is ≥768px, then the three-column Kanban layout must be displayed.

---

### L2-041: Large and Extra-Large Viewport Layout (≥992px)
**Traces to:** L1-009

Screens must take full advantage of large (≥992px) and extra-large (≥1200px) viewports without becoming sparse or oversized.

**Acceptance Criteria:**
1. Given any application screen, when the viewport is ≥1200px, then the maximum content width must be capped (e.g., 1440px) so that line lengths remain readable and content does not stretch edge-to-edge.
2. Given the dashboard, when the viewport is ≥992px, then the dashboard grid must support at least 12 columns to allow rich widget arrangements.

## Theming

### L2-042: Default Dark Monochromatic Theme
**Traces to:** L1-010

The application must apply a single dark, mostly monochromatic theme by default to every screen, component, and chart, with no light-mode option.

**Acceptance Criteria:**
1. Given any screen, when rendered, then the page background must be a dark surface (luminance ≤ 20%) and text must be a near-white or light grey (luminance ≥ 80%) yielding a text-to-background contrast ratio ≥ 7:1 (WCAG AAA for body text).
2. Given any Angular Material component (buttons, dialogs, form fields, menus, snackbars), when rendered, then it must use the dark monochromatic theme palette and must not display any default Material primary/accent colors that fall outside the palette.
3. Given any Chart.js chart, when rendered, then chart background, gridlines, axis labels, and series colors must be drawn from the dark monochromatic palette and must remain legible against the dashboard surface.

## Search

A global search affordance must be accessible from every screen and must search contacts, partners, hackathons, and team members in a single action.

Given any authenticated screen, when the user opens the global search affordance and enters a term of 2 or more characters, then the system must return matching results across contacts, partners, hackathons, and team members, grouped by entity type, within 500 ms for a dataset of up to 10,000 records per type.

## Logging and Observability

### L2-044: Backend Structured Logging
**Traces to:** L1-012

The backend must produce structured JSON logs for every HTTP request, response, exception, authorization decision, and significant domain event, with consistent fields (timestamp, level, correlation ID, user ID where applicable, event name, and event-specific properties).

**Acceptance Criteria:**
1. Given any HTTP request to the backend, when the request is processed, then the backend must log a single structured entry containing timestamp, correlation ID, HTTP method, path, status code, response time, user ID (if authenticated), and remote IP.
2. Given any unhandled exception in the backend, when caught by the error middleware, then the backend must log the exception with full stack trace, correlation ID, and user ID, must return a generic 500 response that does not leak the stack trace, and must increment an error counter.
3. Given any role assignment change, sign-in, sign-out, or delete operation, when it occurs, then the backend must produce an audit log entry of category "audit" with actor user ID, target ID, action, and timestamp.

---

### L2-045: Frontend Logging
**Traces to:** L1-012

The frontend must capture and forward to the backend any unhandled error, failed XHR, and significant user-facing failure, with the same correlation ID used by the originating request when available.

**Acceptance Criteria:**
1. Given an unhandled error in the Angular application, when it is thrown, then the frontend's global error handler must forward a structured log to a backend logging endpoint within 5 seconds, including error message, stack trace, route, user ID (if known), and a generated correlation ID.
2. Given an HTTP error response from the backend, when received by the Angular HTTP interceptor, then the interceptor must log the error with status, URL, correlation ID (echoed from response header), and request body where it does not contain secrets.
3. Given any submitted log from the frontend, when received by the backend, then the backend must accept it, validate it does not contain credentials or session tokens, and persist it with a "frontend" source tag.

## Performance

### L2-046: Page Load Performance
**Traces to:** L1-013

Every primary screen must load within defined performance budgets on a typical 4G mobile network and a mid-range device.

**Acceptance Criteria:**
1. Given a cold load of any primary screen (sign-in, dashboard, contact list, partner list, partner board, hackathon list, team) on a simulated Slow 4G profile and a Moto G4-class device, when the screen is requested, then Largest Contentful Paint must occur within 2.5 seconds and Time to Interactive within 3.5 seconds.
2. Given a warm load (cached) of any primary screen on the same profile, when the screen is requested, then Largest Contentful Paint must occur within 1.5 seconds.
3. Given the production build, when measured by Lighthouse on a representative environment, then the performance score must be ≥ 80 and the accessibility score must be ≥ 95.

### L2-047: API Response Time
**Traces to:** L1-013

Backend API endpoints must respond within defined latency budgets at expected load.

**Acceptance Criteria:**
1. Given a backend running at expected load (up to 50 concurrent users per team, up to 50 teams), when any GET endpoint is called, then the 95th percentile response time must be ≤ 300 ms and the 99th percentile must be ≤ 800 ms.
2. Given the same load, when any POST/PUT/DELETE endpoint is called, then the 95th percentile response time must be ≤ 500 ms and the 99th percentile must be ≤ 1200 ms.
3. Given a search endpoint over a dataset of 10,000 records per type, when called, then the 95th percentile response time must be ≤ 500 ms.

SignalR-pushed messages must reach connected clients within a defined latency budget.

**Acceptance Criteria:**

## Data Protection and Application Security

### L2-049: Encryption in Transit
**Traces to:** L1-014

All client-server communication, including HTTP requests and SignalR connections, must be encrypted using TLS 1.2 or higher.

**Acceptance Criteria:**
1. Given any production environment, when an HTTP request is made to a non-TLS endpoint, then the server must respond with a redirect to the equivalent HTTPS endpoint or refuse the request.
2. Given any TLS connection to the production backend, when the connection is negotiated, then the negotiated protocol must be TLS 1.2 or 1.3 and the cipher suite must come from a documented allowlist of strong suites.

---

### L2-050: Password Storage
**Traces to:** L1-014

Passwords must be stored only as salted hashes using a memory-hard algorithm.

**Acceptance Criteria:**
1. Given a user registration or password change, when the password is persisted, then the stored value must be the output of Argon2id (or PBKDF2-SHA256 with at least 600,000 iterations) with a unique per-user salt; the plaintext password must never be stored or logged.
2. Given the user database, when inspected, then no column or log entry may contain a plaintext password, an unsalted hash, or a hash from a fast algorithm such as MD5, SHA-1, or unsalted SHA-256.

---

### L2-051: Input Validation
**Traces to:** L1-014

Every API endpoint must validate every incoming field against an explicit schema (length, type, format, allowed values), and the frontend must perform the same validations before submission.

**Acceptance Criteria:**
1. Given any backend endpoint, when an input fails validation, then the response must be 400 Bad Request with a structured error body listing field names and validation messages, and the validation must occur before any business logic or database access.
2. Given an oversized input (e.g., a 10 MB note body or a 10,000-character display name), when submitted, then the backend must reject it before reading the full body where possible (e.g., via request size limit).

---

### L2-052: Injection Prevention
**Traces to:** L1-014

The backend must prevent SQL injection, NoSQL injection, command injection, and LDAP injection by using parameterized queries, ORM bindings, and never concatenating user input into queries or shell commands.

**Acceptance Criteria:**
1. Given a code review or static analysis run, when any query in the backend is inspected, then no string concatenation of user input into a SQL or NoSQL query expression must be present.
2. Given a fuzzing test that submits common injection payloads (single quote, drop table, $where, OR 1=1, ${jndi:…}) to every text input field, when run, then the backend must respond with 400 or normal handling and must never expose database errors or execute injected payloads.

---

### L2-053: Output Encoding and XSS Prevention
**Traces to:** L1-014

The frontend must rely on Angular's default contextual output encoding and must never bypass it (e.g., via `[innerHTML]` with untrusted input, `bypassSecurityTrust*`).

**Acceptance Criteria:**
1. Given a code review, when the Angular templates and TypeScript code are inspected, then no use of `bypassSecurityTrust*` on untrusted input must be present, and `[innerHTML]` must only be used with sanitized or hard-coded content.
2. Given a stored XSS test where a malicious script payload is saved as a contact note, partner note, or display name, when that field is rendered on any screen, then the script must not execute and the payload must be displayed as escaped text.

---

### L2-054: CSRF Protection
**Traces to:** L1-014

The backend must protect state-changing endpoints against cross-site request forgery using anti-CSRF tokens or strict SameSite cookies plus an appropriate header check.

**Acceptance Criteria:**
1. Given any state-changing endpoint (POST, PUT, PATCH, DELETE), when the request lacks a valid CSRF token or originates from a cross-site referer, then the backend must reject the request with 403 Forbidden.
2. Given the SameSite=Lax (or Strict) attribute on the session cookie, when a third-party site initiates a top-level GET, then the cookie behavior must be predictable per the SameSite policy and not enable a CSRF flow.

---

### L2-055: Rate Limiting
**Traces to:** L1-014

The backend must enforce rate limits on authentication and other sensitive endpoints to mitigate brute-force, credential-stuffing, and abuse.

**Acceptance Criteria:**
1. Given the sign-in endpoint, when more than 10 requests are received from a single IP within 1 minute, or more than 5 failed attempts target a single account within 15 minutes, then the endpoint must respond with 429 Too Many Requests and log the throttling event.
2. Given the password recovery endpoint, when more than 3 requests target the same email within 15 minutes, then the endpoint must continue to return the generic message but must not send additional emails until the window resets.

---

### L2-056: Secrets Management
**Traces to:** L1-014

The application must never embed secrets (API keys, database passwords, signing keys) in source code, build artifacts, or client bundles.

**Acceptance Criteria:**
1. Given a build of the production frontend bundle, when the bundle is inspected, then it must contain no secrets, environment-specific connection strings, or signing keys.
2. Given the backend at runtime, when configuration is loaded, then secrets must be loaded from environment variables or a secrets store, not from a checked-in configuration file.

**Traces to:** L1-015

Every interactive control must be reachable and operable via keyboard alone, with a visible focus indicator and a logical tab order.

Every interactive control, image, and dynamic region must expose accurate semantic labels and ARIA attributes for screen readers.

## Component and API Architecture

### L2-060: Reusable Component Library
**Traces to:** L1-016

The Angular workspace must include a `components` library that exports the application's branded presentation components — buttons, error messages, notifications, dialogs, form fields, snackbars — each implemented as a thin wrapper over an Angular Material primitive and themed to the dark monochromatic palette.

**Acceptance Criteria:**
1. Given any feature library, when it imports a button, dialog, error message, notification, or form field, then it must import from the `components` library and not directly from `@angular/material`.
2. Given the `components` library, when its public API is inspected, then it must export all reusable UI components used by feature libraries and must not export feature-specific components.

---

### L2-061: API Library with Injection Tokens
**Traces to:** L1-017

The Angular workspace must include an `api` library that defines TypeScript models for every backend resource and exposes service contracts via injection tokens, so that feature libraries depend on the contracts rather than concrete service classes.

**Acceptance Criteria:**
1. Given any feature library, when it consumes a backend API, then it must inject the relevant service via an injection token defined in the `api` library and must not import a concrete service class directly.
2. Given the `api` library, when its public API is inspected, then it must export models, injection tokens, and contract interfaces; concrete service implementations must also live in `api` but be provided to consumers only through the tokens.
3. Given a unit test for any feature library, when the test runs, then it must be able to substitute a mock implementation of any API service via the injection token without changing the feature code.

---

### L2-062: Backend CQRS with MediatR
**Traces to:** L1-017

The backend must implement a command and query separation pattern using the free version of MediatR, where every controller action dispatches a single command or query rather than calling business logic directly.

**Acceptance Criteria:**
1. Given any controller action, when its body is inspected, then it must dispatch a single MediatR request and return the result, with no inline business logic, no direct repository access, and no direct service composition.
2. Given any command or query, when defined, then it must have a request type, a response type, and a single handler class.
